1. Introduction to AMBA AHB

The Advanced Microcontroller Bus Architecture (AMBA) is a set of specifications developed by ARM. Its primary goal is to standardize communication between different functional blocks of a System-on-Chip (SoC). Think of an SoC as a bustling city; AMBA provides the rules of the road so that traffic (data) flows smoothly without collisions.

On-Chip Protocols

These act as ultra-fast internal highways connecting the CPU and memory inside a single silicon chip. They focus on moving massive amounts of data instantly over microscopic distances. Examples include AHB and AXI.

Peripheral Protocols

These act as external bridges connecting the main chip to outside components like sensors, keyboards, and screens. They focus on using fewer wires to reliably send data across a physical circuit board. Examples include I2C, SPI, and USB.

The AMBA Family Protocols

  • AHB (Advanced High-performance Bus): The high-speed backbone. It is designed for high-bandwidth communication. It features single clock-edge operation, burst transfers, and a pipelined structure.
  • APB (Advanced Peripheral Bus): The low-speed local road. Used for connecting low-power, low-bandwidth peripheral devices (like timers or UARTs). You need a "Bridge" component to convert fast AHB signals into slower APB signals.
  • AXI (Advanced eXtensible Interface): The next-generation upgrade over AHB, used for extreme high-performance, low-latency, and multi-channel applications.

2. AHB System Architecture

The AHB protocol defines the interaction between three main components. No component is permanently locked into a single role forever; roles can be interchanged depending on the transaction needs.

Visualize the Architecture

Imagine a central hub (the Interconnect) with multiple paths radiating outward. On the left side, you have the Managers (Masters)—such as the CPU or a DMA controller. They are the "brains" that initiate requests. On the right side, you have the Subordinates (Slaves)—such as RAM, ROM, or external interfaces. They are the "workers" waiting for instructions.

Between them lies a thick bundle of wires. When a Master wants to talk to a Slave, it shouts the Slave's address into the Interconnect. The Interconnect acts as a smart switchboard, decoding the address, lighting up a specific "Select" wire (HSEL) for the target Slave, and bridging the connection.

  1. Managers (Masters): Initiate transactions by issuing read and write commands. They drive the address, control signals, and write data. To avoid chaos, only one Manager can actively control the bus at any given microsecond.
  2. Subordinates (Slaves): Respond to the Manager's requests. If a Manager sends an address that doesn't match any Subordinate on the network, the Interconnect generates an "Error" response.
  3. Interconnects (Decoders & Multiplexors): Handle the routing. The Decoder looks at the Master's address and activates the correct Slave. The Multiplexor routes the chosen Slave's read data back to the Master.

3. Key AHB Signals

AHB operates synchronously. Every event happens exactly on the rising edge of the system clock (HCLK), ensuring highly predictable timing.

Signal Group Signal Name Beginner's Explanation
Global Signals HCLK The heartbeat of the system. Everything syncs to its rising edge.
HRESETn Active-low reset. '0' wipes the system clean; '1' means normal operation.
Address & Control HADDR[31:0] The destination address. "Where am I writing to, or reading from?"
HWRITE The direction switch. '1' = Write (Master to Slave). '0' = Read (Slave to Master).
HSIZE[2:0] The size of the data packet. Determines if we are sending 8 bits (Byte), 16 bits (Half-word), or 32 bits (Word).
HBURST[2:0] The grouping rule. Tells the slave "I am sending 1 item" or "I am sending a burst of 4 items in a row."
HTRANS[1:0] The state of the transfer. Tells the slave if the master is IDLE, BUSY, starting a new transfer (NONSEQ), or continuing a burst (SEQ).
Data Buses HWDATA[31:0] Write data highway. Carries data from Master to Slave.
HRDATA[31:0] Read data highway. Carries data from Slave to Master.
Transfer Response HREADY The "Wait a second!" signal. Driven by the Slave. '1' means "I'm ready, let's go." '0' means "Hold on, give me another clock cycle."
HRESP The transaction receipt. '0' means the transfer was successful (OKAY). '1' means something went wrong (ERROR).

4. Pipelining & Basic Transfers

The defining feature of AHB is that it is a pipelined protocol. This means a basic transfer is split into two distinct phases that overlap for maximum efficiency.

  • Address Phase (1st cycle): The Master places the target address and control signals (like Read/Write, Size) onto the bus. This takes exactly one clock cycle.
  • Data Phase (2nd cycle): The actual data is transferred (either written by Master or read from Slave). This phase can take one cycle, or it can be extended if the Slave needs more time.
Visualize the Pipeline (The Restaurant Analogy)

Imagine a fast-food drive-thru with two windows: an Order Window (Address Phase) and a Pickup Window (Data Phase).

Car A pulls up to the Order Window at 12:00 PM and places an order (Address Phase). At 12:01 PM, Car A moves to the Pickup Window to receive its food (Data Phase). But crucially, at that exact same moment (12:01 PM), Car B pulls into the Order Window to place its order.

Because of pipelining, the Data Phase of Transfer 1 overlaps perfectly with the Address Phase of Transfer 2. The bus is never sitting idle!

Wait States (The HREADY Signal)

What happens if the kitchen is slow and Car A's food isn't ready? The cashier tells Car A to wait. This is exactly what the HREADY signal does.

  • If HREADY = 0, the slave is inserting a "Wait State". The Data Phase is extended by an extra clock cycle.
  • During a Write: If the slave says "wait", the master must freeze and hold its data steady on the HWDATA bus until HREADY becomes 1.
  • During a Read: If the slave says "wait", the master knows that whatever is currently on the HRDATA bus is garbage. The master will only sample the data when HREADY finally becomes 1.
  • Crucial Pipeline Rule: If the Data Phase of Transfer 1 is extended, the Address Phase of Transfer 2 is also frozen in place. Everything waits together.

5. Transfer Types (HTRANS)

The 2-bit HTRANS[1:0] signal acts as the Master's status indicator, telling the Slave exactly what is happening in the current cycle.

Value Type Meaning & Behavior
00 IDLE The Master is doing nothing. Even if there are random addresses on the bus, the Slave must ignore them and provide a standard "OKAY" response.
01 BUSY The Master is in the middle of a burst but needs a breather. For example, it needs one clock cycle to fetch the next piece of data internally. The Slave must wait and provide an "OKAY" response.
10 NONSEQ "Non-Sequential". This indicates the very first transfer of a new transaction. The address has no relationship to whatever happened previously.
11 SEQ "Sequential". This indicates the 2nd, 3rd, or 4th transfer in a continuous burst. The slave instantly knows that the address for this transfer is just the previous address plus the data size.

6. Burst Operations (HBURST)

Instead of negotiating a brand new transfer for every single piece of data, a Master can declare a "Burst". A burst is a guarantee: "I am going to send you 4 (or 8, or 16) packets of data back-to-back." This vastly improves throughput. A single packet in a burst is called a beat.

Incrementing Bursts (INCR)

This is straightforward. The address simply steps forward by the size of the data for each beat.

Analogy: Like reading a book left to right. Page 1, Page 2, Page 3, Page 4.

  • If Start Address = 0x20 (and size is 4 bytes).
  • Beat 1 = 0x20
  • Beat 2 = 0x24
  • Beat 3 = 0x28
  • Beat 4 = 0x2C

Wrapping Bursts (WRAP)

Wrapping bursts are primarily used by CPU caches. When a CPU misses a cache, it needs a whole block of memory. It asks for the specific missed address first, then wraps around to fetch the rest of the block.

Analogy: Like a clock dial. You read 10, 11, 12... and then wrap back around to 1.

The Mathematics of Wrapping:
To calculate where a burst wraps, you must find the Address Boundary.

Address Boundary = Number of Beats × Size of Transfer (in bytes)

Example: A 4-beat wrapping burst (WRAP4) of 32-bit words (4 bytes each).
Boundary = 4 beats × 4 bytes = 16 bytes.

If the Master starts the burst at an offset address like 0x34:
1. Beat 1: 0x34
2. Beat 2: 0x38
3. Beat 3: 0x3C
4. Beat 4: 0x30 (It hits the 16-byte upper limit and wraps back to the base address of that block!).

7. Subordinate Response (HRESP)

Communication is a two-way street. After the Master initiates the transfer, the Slave must report back on the status using the 1-bit HRESP signal.

  • 0 = OKAY: The transfer is proceeding normally. It has either completed successfully, or the slave is simply asking for more time (pending via HREADY = 0).
  • 1 = ERROR: A catastrophic failure. Perhaps the Master tried to write to a Read-Only Memory (ROM), or accessed a protected secure memory region.

Note on Errors: When an ERROR occurs, the Slave must assert it for exactly two clock cycles. Why? Because of pipelining! The Master has already started the next address phase. The two-cycle error gives the Master time to cancel the next overlapping transfer and safely abort.
